# 如何理解HTML语义化

  • 让人更容易读懂（增加代码可读性）
  • 让搜索引擎更容易读懂（SEO）

# BFC(块级格式化上下文)

  • 一块独立渲染区域，内部元素的渲染不会影响边界以外的元素

  • 满足下列的任意一个或多个条件即可

    1. float的值不是none。（float:left 或者float:right）
    2. position的值不是static或者relative。（position:absolute或者position:fixed）
    3. display的值是inline-block、table-cell、flex、table-caption或者inline-flex
    4. overflow的值不是visible（overflow:hidden、overflow:scroll）
    5. 父元素与正常文件流的子元素（非浮动子元素）自动形成一个BFC

  • BFC的特征

    1. 所有子元素（包含浮动元素）与容器（父元素）左边对齐
    2. 属于同一个BFC的父元素和子元素，相邻的父子或者兄弟间margin垂直方向会重叠，若2个元素属于不同的BFC，则垂直方向不会重叠
    3. 可以自动撑开容器（若子元素是float的，父元素设置overflow:hidden，父元素就形成一个BFC)
    4. 2个同级的BFC 不会重叠（下面的例子：用于父div里面有2个div，左边一个浮动，右边一个设置overflow :auto，相当于2个子元素是2个BFC）

# margin 纵向重叠问题

  • 相邻元素的margin-top 和 margin-bottom 会发生重叠 解决办法是让相邻的元素不属于一个BFC

# 子元素设置margin-top，父元素也受影响

  • 根据规范，一个盒子如果没有上补白(padding-top)和上边框(border-top)，那么这个盒子的上边距会和其内部文档流中的第一个子元素的上边距重叠

# margin 负值问题

  • margin-top 和 margin-left 负值，元素向上、向左移动
  • margin-right 负值，右侧元素左移，自身不受影响
  • margin-bottom 负值，下方元素上移，自身不受影响

# 异步

# event loop （事件循环）

  • js 是单线程运行的
  • 异步要基于回调来实现
  • event loop就是异步回调的实现原理

# js 是如何执行的

  • 从前到后，一行一行的执行
  • 如果有报错停止执行
  • 先把同步代码执行完成，再执行异步

# event loop 过程

  • 同步代码，一行一行放在call Stack 执行
  • 遇到异步，会先记录下来，等待时机（定时、网络请求等）
  • 如果call Stack 为空（即同步代码执行完成）
  • 先执行当前的微任务 （如有则移动到micro task queue 执行）
  • 然后尝试DOM渲染
  • 最后触发event loop 轮询查找Callback Queue， 如有则移动到Call Stack 执行

# DOM 事件和event loop 关系

  • DOM 事件也是使用回调，基于event loop 但不是异步的，是靠用户点击

# promise 的then 和 catch 如何影响状态的变化

  • then 正常返回resolved 里面有报错则返回rejected
  • catch 正常返回resolved 里面有报错返回rejected
Promise.resolve().then(() => {
    console.log(1) // 1
    throw new Error('error') 返回一个rejected promise
}).catch(() => {
     console.log(2) // 2 只要没有报错 返回一个 resolve 状态的promise
}).then(() => {
    console.log(3) // console.log(3)
})

# 如何处理异步

  • 异步回调 callback hell
  • Promise then catch 链式调用，但也是基于回调函数
  • async/await 是同步语法，彻底销灭回调函数

# async/await 和 promise 的关系

  • async/await 是 promise的语法糖
  • 执行async 函数，返回的是promise
  • await 相当于 Promise 的then
  • try...catch 可捕获异常，代替了promise的catch

(async function(){

async function fn(){
    return 100

(async function(){
    const a = fn() //返回 resolve 状态的promise

# 宏任务(macroTask) 和 微任务 (microTask)

# 什么是宏任务，什么是微任务

  • 宏任务：setTimeout, setInterval, Ajax, DOM 事件(虽然不是异步，但是也是基于event loop 执行的)
  • 微任务：Promise async / await
  • 微任务执行时机比宏任务要早

# event loop 和 DOM 渲染

  • 每次Call Stack 清空（即每次轮询结束），即同步任务执行完 都是DOM重新渲染的机会，DOM结构如有改变则重新渲染 然后再去触发下次Event Loop

# 微任务和宏任务的区别

  • 宏任务：DOM 渲染后触发，如setTimeout
 $('#container')
    .append('<div>123445666t</div>')
    .append('<div>aaaaaaaaaa</div>')
 let len = $('#container').children().length

     * 宏任务在dom渲染后触发，所以alert阻断代码执行后页面已经渲染完成，
     * 之后执行宏任务setTimeout
    */
     setTimeout(() => {
        console.log('len', len)
        alert(1)
    })

# 为什么微任务比宏任务执行的更早？

  • 综上所述，微任务是在dom更新但未渲染时触发，宏任务是在dom渲染完成后执行，所以 微任务比宏任务先执行

# 宏任务与微任务的根本区别

  • 微任务比宏任务执行早
  • 微任务是ES6语法规定的（例如ajax宏任务 在es6规范中是没有的）
  • 宏任务是由浏览器规定的

# http

# http 常见状态码有哪些

# 状态码分类

  • 1xx 服务器收到请求
  • 2xx 请求成功
  • 3xx 重定向 如 302
  • 4xx 客户端错误 如404
  • 5xx 服务端错误 如 500

# 常见状态码

  • 200 成功
  • 301 永久重定向（配合location， 浏览器自动处理）
  • 302 临时重定向（配合location， 浏览器自动处理）
  • 304 资源未被修改
  • 404 资源未找到
  • 403 没有权限
  • 500 服务器错误
  • 504 网关错误

# http常见的header 有哪些

# Request Headers

  • Accept 浏览器可接收的数据格式
  • Accept-Encoding 浏览器可接收的压缩算法，如gzip
  • Accept-Languange 浏览器 可接收的语言，如zh-CN
  • Connection：keep-alive 一次TCP 连接重复使用
  • cookie 同域会自动携带cookie信息，跨域不会
  • Host 请求的域名
  • User-Agent（简称UA）浏览器信息
  • Content-type 发送数据的格式，如 application/json

# Response Headers

  • Content-type 返回数据的格式，如 application/json
  • Content-length 返回数据的大小，多少字节
  • Content-Encoding 返回数据的压缩算法，如 gzip
  • Set-Cookie 服务端修改cookie

# 自定义header

  • headers:{'X-Request-With': 'XML'} ...

# 缓存相关的headers

  • Cache-Control Expires
  • Last-Modified If-Modified-Since
  • Etag If-None-Match

# Restful API

  • 传统API 设计：把每个url当做一个功能
  • Restful API 设计： 把每个url当做一个唯一的资源

# 如何设计成一个资源

  • 尽量不用url 参数
    1. 传统API设计： /api/list?pageIndex = 2
  • 用method表示操作类型
    1. Restful API设计：/api/list/2

# methods 方法有哪些

  • get 获取数据
  • post 新建数据
  • patch/put 更新数据
  • delete 删除数据

# 用method 表示操作类型（Restful API 设计）

  • post 请求 /api/blog
  • patch 请求 /api/blog/100
  • get请求 /api/blog/100

# http为什么要缓存

  • 让页面加载的更快些
  • disk cache 为本地缓存

# 页面加载更快的核心是什么

  • 网络的时间
  • 网络的不稳定性

# 浏览器http请求缓存流程

# http 缓存 - 强制缓存

  • 初次请求服务器，返回资源和如果服务端觉得需要缓存会加个Cache-Control 浏览器发现有Cache-Control 需要缓存 浏览器会把资源缓存 下来，如果 浏览器再次请求发现Cache-Control 并且还在有效期，浏览器会从本地缓存中查找 不会去请求服务器资源，从而加快访问速度，如果再次请求，发现缓存过期，浏览器 会重新请求服务端，返回相应的资源和Cache-Control

# cache-control

  • cache-control 是在 Response Headers 中由服务端添加，值如下：
  • max-age 设置最大缓存时间 例如：Cache-Control:max-age=3153600(单位是秒)缓存一年
  • no-cache 不用本地缓存，正常的像服务端发起请求
  • no-store 不用本地缓存而且不用服务端的一些缓存措施
  • private 允许最终用户缓存
  • public 允许中间的一些路由，代理也可以做一些缓存

# Expires

  • 同在Response Headers 中
  • 同为控制缓存过期
  • 已被 Cache-Control 代替

# http 缓存-协商缓存（对比缓存）

  • 服务端缓存策略 （服务端来判断资源是不是可以被缓存）
  • 服务端判断客户端资源，是否和服务端资源一样 一致则返回304，否则返回200和最新的资源

# 服务端如何判断客户端资源是否和服务端资源一样

  1. 初次请求，服务端返回资源和资源标识
  2. 再次请求，带着资源标识，服务端根据标识进行判断
  3. 返回304，或返回资源和新的资源标识

# 资源标识

  • 在Response Headers 中，有两种：

  1. Last-Modified 资源的最后修改时间

    • 初次请求服务器，返回资源和Last-Modified
    • 再次请求服务器，Request Headers 带着 If-Modified-Since 值为Last-Modified的值
    • 服务器根据Last-Modified的值判断修改时间，返回304，或者返回资源和新的Last-Modified

  2. Etag 资源的唯一标识

    • 初次请求服务器，返回资源和Etag
    • 再次请求服务器，Request Headers 带着 If-None-Match值为Etag的值
    • 服务器对Etag 进行对比，返回304或者新的资源和新的Etag

# Last-Modified和Etag 对比

  • 会优先使用Etag
  • Last-Modified 只能精确到秒级
  • 如果资源被重复生成，而内容不变，则Etag更精确

# 刷新页面对http缓存的影响

  • 正常操作：强制缓存有效，协商缓存有效
  • 手动刷新：强制缓存失效，协商缓存有效
  • 强制刷新：强制缓存失效，协商缓存失效
